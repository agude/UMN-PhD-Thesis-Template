\begin{abstract}
We describe the software architecture used to deploy the Aframe and AMPLFI algorithms, an end to end machine-learning based system for real time detection and parameter estimation of gravitational wave data. We detail the production deployment of Aframe and AMPLFI during the the third part of the LIGO-Virgo-KAGRA collaboration's  fourth observing run, and discuss the system's contributions to public alerts. 

\end{abstract}

\section{Introduction}
The first discovery of gravitational waves (GWs) from a binary black hole (BBH) merger in 2015 jump-started the era of observational GW science \cite{Abbott_2016}. The LIGO-Virgo-KAGRA (LVK) collaboration's third observing run (O3) was the first which distributed real-time public alerts to the astronomical community. Since then, hundreds of real-time candidate events have been made public\footnote{//gracedb.ligo.org/superevents/public}. In May 2023, the LVK began its fourth observing run (O4) ending in November 2025. Already, the LVK has published GWTC-4.0 consisting of \red{128} GW candidate events from just the first part of the fourth observing run (O4a). As the Advanced LIGO \cite{buikema2020sensitivity, Aasi_2015}, Advanced Virgo \cite{Acernese_2015}, and KAGRA \cite{KAGRA:2020tym} interferometers sensitivity increase, so does the quantity of detected events. 

Several pipelines operated real-time searches during O4. The GstLAL, MBTA, PyCBC, and SPIIR pipelines are all matched-filtering based searches targeting compact binary coalescences (CBCs) \cite{PhysRevD.108.043004, Allene_2025, Dal_Canton_2021, PhysRevD.105.024023}. Also operating were minimally modeled pipelines cWB \cite{PhysRevD.93.042004, PhysRevD.105.083018}, MLy \cite{PhysRevD.110.104034}, and oLIB \cite{Lynch:2015yin} which are designed to be sensitive to a broader range of astrophysical sources. 


Over the past decade, machine-learning (ML) research in gravitational-wave (GW) astronomy has emerged as a flourishing  field \cite{Cuoco_2020}. Non-linear noise sources, availability of robust numerical relativity simulations, and strict requirements of real-time analysis make ML solutions highly compelling candidates for next generation data analysis methods. The ability to operate in real-time on accelerated hardware like GPUs provides reduced alert latencies, and smaller inference compute footprints compared with CPU-based filtering methods. In addition, ML methods provide a pathway to handle the growing computational burdens associated with the growing detection rate \cite{gwtc-4}. Despite rapid growth in GW ML literature, these methods have seen limited adoption by the LVK for collaboration analyses. The process of moving ML based research beyond publications and into production-ready systems is critical for proving their utility, and, more importantly, identifying where they can improve. 

In previous works, Aframe, a ML based detection algorithm, was introduced and used to search for GW signals from binary black hole (BBH) mergers in O3 \cite{aframe-methods, marx2025machine}. In parallel, AMPLFI, a parameter estimation (PE) method based on likelihood-free inference was developed and validated as a method for producing real-time alert data products \cite{amplfi, Marx:2025ioo}. These algorithms were designed to operate in unison as an end-to-end real-time search pipeline. In this work, we describe the software system used to deploy Aframe and AMPLFI for real-time analysis during O4c. In Sec.~\ref{sec:models}, we review the Aframe and AMPLFI algorithms. In Sec.~\ref{sec:event-identification} we discuss how Aframe loads, validates, and processes strain data to identify events. In Sec.~\ref{sec:data-products} we detail the methodology for producing the suite of low-latency alert data products from the outputs of Aframe and AMPLFI. In Sec.~\ref{sec:software-arch} we show how our software architecture design minimizes latency. Finally, in Sec.~\ref{sec:o4c} we analyze results from the real time analysis of O4c.


\section{Aframe and AMPLFI}  \label{sec:models}
Here we briefly describe the Aframe detection and AMPLFI parameter estimation models for CBC sources. For full details, we refer readers to Refs.~\cite{aframe-methods, Marx:2025ioo}. We also describe the datasets used to train the models deployed during O4c. 

\subsection{Aframe}
Aframe is a deep-learning-based detection algorithm designed to identify GW signals from well-modeled astrophysical sources. In particular, Aframe has been trained and used to search for GW signals from binary black hole (BBH) coalescences. The core architecture utilizes a 1D convolutional neural network (CNN) that processes strain data from multiple detectors. Aframe analyzes 1-second windows of whitened strain data sampled at $2048$ Hz from the Hanford and Livingston LIGO interferometers. The neural network is trained to map this input to a single scalar detection statistic that indicates how likely the algorithm finds a signal present in the input data segment. Aframe's analysis window strides across the data at a fixed rate called the \emph{inference sampling rate}, producing a time-series of neural network outputs. The inference sampling rate determines the precision with which the coalescence time of the merger can be estimated. For O4c online analysis, the inference sampling rate is 512 Hz. The prior used to train Aframe for O4c deployment is listed in Table \ref{table:aframe-priors}, which is the same as that used in previous works. 

\subsection{AMPLFI}
After candidate events are identified, parameter estimation (PE) is performed by AMPLFI, a ML framework based on likelihood-free inference (LFI) \cite{amplfi, Marx:2025ioo}. In particular, AMPLFI uses a normalizing flow neural-network architecture which is trained to model the Bayesian posterior distribution, $p(\theta\lvert d)$ of source properties $\theta$ given data $d$. The model uses time and frequency representations of the strain data along with an estimate of the power spectral density (PSD) surrounding the trigger identified by Aframe as contextual input to condition the normalizing flow. 

AMPLFI models are trained for fixed detector configurations. To maximize sky localization precision, two distinct AMPLFI models for Hanford-Livingston (HL) and Hanford-Livingston-Virgo (HLV) detector configurations are trained. The online deployment dynamically selects the appropriate model, prioritizing the HLV model when science-quality data from the Virgo interferometer at the time of the event is available. The signal priors used to train the AMPLFI models used during O4c is listed in Table \ref{tab:amplfi-priors}. Compared with Refs.\cite{amplfi, Marx:2025ioo}, the chirp mass prior is lowered to 5 $\mathrm{M}_{\odot}$ motivated by analyzing mock data challenges with Aframe which demonstrated sensitivity to these signals. In addition, we extend the distance prior from 3 to 6 Gpc to account for increased detector sensitivity between O3 and O4, and to reduce instances where AMPLFIs luminosity distance posteriors would rail against the prior boundary.

\section{Event Identification} \label{sec:event-identification}
In this section, we detail how Aframe streams, loads, processes, and analyzes GW data to identify events, and how AMPLFI performs PE followup.


\subsection{Frame Delivery and Loading}
GW data in the LVK are currently distributed in real time via 1 second chunks known as ``frames''. Online search pipelines must aggregate and process these frames into the appropriate format for their analysis. The Aframe data loader continuously queries designated directories on disk for incoming real-time frame files. If a required frame does not appear within a specified timeout period (typically $\sim10$ s), the data loader will yield a timeout signal and dynamically reset its internal clock to query the latest available data. This ensures that transient frame delivery issues will not cause the data loader to halt indefinitely. In addition, the data loader employs automated retry logic to catch and recover from common I/O errors, such as corrupted file headers or incomplete frame writes.

This current system for delivering GW frames introduces a fundamental latency due to the discrete frame size. Delivery of frames as discrete 1 second chunks means pipelines must wait for the whole frame to be complete before analysis. In the worst case, a GW event occurs at the beginning of a frame, and downstream tasks must wait for the remaining full second of data before proceeding. For 1 second frames, this \emph{discrete frame latency} adds an additional 0.5 seconds on average. See Figure \ref{fig:discrete-frame-latency} for an illustration. Next generation data delivery (NGDD) systems are being developed for use in future observing runs that will deliver data in 1/16 s increments, significantly reducing these incurred latencies. In addition, the Aframe/AMPLFI pipeline deployed at the LIGO Caltech computing cluster must wait for the data to be transferred from the interferometer sites to Caltech before analysis. This transfer latency typically takes $\sim 1$ s.

\subsection{Data Quality}
Data quality information for each interferometer is distributed in the same frame files as the GW strain via a separate time-series channel, known as a state vector. The state vector is a time-series of bit masks where each bit ``flags" whether a specific quality of the detector is active. Most important for real-time GW searches are the first two bits 

\begin{itemize}
    \item \texttt{HOFT\_OK}: Indicates that the strain data was processed correctly
    \item \texttt{OBSERVATION\_INTENT} Indicates the interferometer was placed in observation mode
\end{itemize}


State vectors are typically sampled at 16 Hz. When processing a new frame, Aframe checks that both the \texttt{HOFT\_OK} and \texttt{OBSERVATION\_INTENT} flags are active for both Hanford and Livingston. If either flag is not active at any point in the frame for either interferometer, the frame is not considered ``analysis ready". To perform inference Aframe requires at least 64 continuous seconds of analysis ready data. This ensures there is enough data to estimate the PSD for whitening. When a frame is not analysis ready, the input strain data and neural network output buffers are reset. Inference will restart once 64 seconds of continuous analysis ready data is accumulated. This PSD ``burn-in" period makes $\sim 1$ minute of data at the beginning of each coincident HL science-quality segment not analyzeable. In the future, using a PSD estimate from a previous segment, or using a smaller amount of data to estimate the PSD will increase the quantity of analyzable data. 

\subsection{Resampling}
Delivered frames are sampled at 16 kHz. As a reminder, Aframe and AMPLFI are trained using data sampled at 2048 Hz, well above the Nyquist frequency for even the lowest mass BBH mergers in the prior ranges (see Tables \ref{tab:amplfi-priors} and \ref{table:aframe-priors}). So, the frames must be resampled at 2048 Hz in real time. Resampling methods that utilize finite impulse response filters corrupt portions of the beginning and end of the time-series due to filter settle in. To mitigate these edge effects, the Aframe data loader maintains a 2 second buffer of the most recent frames in memory. When a new frame arrives, it is appended to the buffer. The now 3 second frame buffer is resampled to the desired sample rate. The uncorrupted middle second of data is appended to a separate buffer which maintains 64 seconds of resampled data for analysis by Aframe. An illustration of this process is shown in Figure \ref{fig:resampling}. Because resampling the frame at time T requires the frame at time T + 1, this procedure incurs a 1 second latency. When NGDD systems begin to deliver frames in smaller chunks, this inherent latency from waiting for the next frame will be reduced.

\subsection{Whitening}
Before Aframe performs inference, the strain data is filtered and then whitened, which requires an estimate of the PSD. During training, the PSD is estimated using the previous 8 seconds of data for each sample. This PSD duration is limited by GPU memory constraints when training with large batch sizes. During real time processing, GPU memory is no longer a bottleneck, so a longer segment of 64 seconds  can be used to estimate the PSD more accurately. To facilitate this, a 64 second buffer of resampled data is kept in GPU memory. When the latest resampled frame is complete, it is moved to the GPU and appended to the front of the buffer. Overlapping windows of data, with time deltas corresponding to the inference sampling rate, are "unfolded" into a batch. The PSD estimated from the preceding 64 seconds of data is used to whiten each batch element. The whitened batch is passed to Aframe for inference. This procedure is illustrated in Figure~\ref{fig:whitening}.

\subsection{Inference}
After a batch of windows is whitened, they are analyzed by Aframe producing a time-series of detection statistics. The time-series of detection statistics is then appended to an output buffer which contains the most recent 8 seconds of detection statistics. This time-series is then integrated using a top-hat filter to produce a time-series of integrated detection statistics. Events are identified by finding peaks in the integrated time-series. An overview of this procedure is shown in Figure \ref{fig:inference}. 

Accurately estimating the significance of an event requires identifying the peak in the integrated network output. If the coalescence time of the event lies at the end of the frame, the integrated network output will not reach its peak until the next frame. In these cases, we wait for inference outputs from the next frame, which contains the integrated network output peak. See Fig.~\ref{fig:ramp-up} for an illustration.

\subsection{Parameter Estimation} \label{sec:amplfi}
Once a significant event is identified, AMPLFI is immediately triggered to produce posterior samples for the source parameters we need to estimate. While Aframe only analyzes Hanford and Livingston data for detection, the inclusion of Virgo data for parameter estimation leads to significantly smaller localization areas \cite{PhysRevLett.119.141101}. So, two AMPLFI models, one trained using a Hanford-Livingston (HL) configuration, and the other using a Hanford-Livingston-Virgo (HLV) are utilized. If science quality Virgo data is available at the time of an event, the HLV AMPLFI model will be used. Otherwise, the HL AMPLFI model will be used.

\section{Data Products} \label{sec:data-products}
Several data products are required to be uploaded to the Gravitational-Wave Candidate Event Database (GraceDB) for a pipeline to contribute to public alerts. These data products provide critical information for effective EM followup \cite{Chaudhary:2023vec}. In this section, we provide an overview of the pipelines methodology for producing these data products from the outputs of Aframe and AMPLFI. Later, in Sec.~\ref{sec:software-arch} we discuss how data product computation is distributed among several sub-processes to minimize time-to-alert. 

\subsection{False Alarm Rate}
The statistical significance of an event is one of the most important factors for informing EM followup strategy. One measure used in GW data analysis is the false alarm rate (FAR) of an event, which indicates how often a random coincidence will produce an event of equal or greater significance. Events with FARs less than one per seven months ($< 5.51 \times 10^{-8}$ ) qualify as significant, are made publicly available, and are followed up with PE and additional human vetting\footnote{See \url{https://emfollow.docs.ligo.org/userguide/analysis} for more information on alert thresholds}. Aframe, like many other detection methods, estimates FAR using the time-slide technique \cite{Aubin_2021, Dal_Canton_2021}. For an event with detection statistic $x$, the FAR of an event can be estimated by counting the number of background events with a equal or greater detection statistic, $N_b(x)$ and dividing by the analyzed background live time $T_b$

\begin{equation}
    \mathrm{FAR}(x) = \frac{N_b(x)}{T_b}
\end{equation}
\noindent
To estimate $N_b(x)$, Aframe analyzed 100 years of time-slide data from the O4b observing run. During the O4c deployment, Aframe used a static noise distribution to estimate FAR's. In the future, analyzing additional time-slides as data is accumulated during the run can account for changing noise in the detectors over time.


\subsection{Source Classification}
An event FAR does not include prior knowledge about the astrophysical population of sources. For example, it is known that high mass ratio mergers are more rare than equal mass ratio mergers, so at a fixed FAR, a 30 $\mathrm{M}_{\odot}$ + 30 $\mathrm{M}_{\odot}$ merger is less likely to be astrophysical than a 10 $\mathrm{M}_{\odot}$ + 100 $\mathrm{M}_{\odot}$ merger. The probability of astrophysical origin ($p_{\mathrm{astro}}$) is a detection statistic that accounts for prior knowledge on the rates of events across the parameter space. For a detailed treatment of Aframe's method for estimating $p_{astro}$, see Appendix A in Ref.~\cite{aframe-methods}. Briefly, Aframe constructs a distribution of detection statistics by analyzing $\sim$ 500,000 simulated signal injections from an astrophysical population motivated by observations from previous observing runs. Then, assuming some astrophysical merger rate, a probability of astrophysical origin can be estimated. 


Currently, Aframe's $p_{\mathrm{astro}}$ model does not incorporate source property information beyond the events significance. So, events with the same FAR will have the same $p_{\mathrm{astro}}$ estimate regardless of their source properties, which is undesirable. Matched-filtering based searches can incorporate source property information from the best fit template into their $p_{astro}$ models. While Aframe does not produce source parameter estimates from templates, AMPLFI provides full posteriors on source properties for use in significance estimation and source classification. So, the probability $p_{\alpha}$ of a source belonging to category $\alpha \in \{\mathrm{BBH}, \mathrm{BNS}, \mathrm{NSBH}\}$ can be calculated by integrating the AMPLFI posterior over the appropriate region $R_{\alpha}$

\begin{equation}
    p_{\mathrm{\alpha}} = p_{\mathrm{astro}}\int_{R_{\alpha}} p(\theta |d) d\theta
\end{equation}

with regions

\begin{align}
    R_{\mathrm{BBH}} = \{ \theta : m_1 > 3, m_2 > 3\} \\
    R_{\mathrm{BNS}} = \{ \theta : m_1 <3, m_2 < 3\} \\ 
    R_{\mathrm{NSBH}} = \{ \theta: m_1 > 3, m_2 < 3\}
\end{align}

where $m_1 > m_2$ by definition. 

\subsection{EM-Bright}
The LVK does not currently release full posterior distributions for events in low- or medium-latency. Instead, the following physically motivated probabilities (known as EM-Bright) are distributed:

\begin{itemize}
    \item HasNS: the probability that the source contains at least one neutron-star (NS)
    \item HasRemnant: the probability that remnant matter is left after the merger 
    \item HasMassGap: the probability that one of the objects is between 3 and 5 $\mathrm{M}_{\odot}$
\end{itemize}

For matched-filtering pipelines, a ML-based classifier models these probabilities using the best-fit template parameters \cite{Chatterjee:2019avs}. Aframe does not directly produce template parameters. Instead, EM-Bright quantities are derived from posterior samples generated by AMPLFI. Each quantity can be estimated by integrating the posterior over the appropriate region of parameter space, $R_X$

\begin{equation}
    p(X) = \int_{R_X} p(\theta |d) d\theta \approx \frac{1}{N}\sum_{i=1}^{N}\mathbf{1}_{R_X}(\theta^{(i)})
\end{equation}

\noindent where $N$ is the number of posterior samples produced by AMPLFI, $\theta^{(i)}$ is the $i^{th}$ sample, and $\mathbf{1}_{R_X}$ is the indicator function over region $R_X$. The regions are defined as

\begin{align}
    R_{\mathrm{HasNS}} &= \{ \theta \mid m_2 \leq 3M_{\odot} \} \\
    R_{\mathrm{HasMassGap}} &= \{ \theta \mid \exists \, m_i \in [3M_{\odot}, 5M_{\odot}] \} \\
    R_{\mathrm{HasRemnant}} &= \{ \theta \mid \text{Masses and spins allow tidal disruption} \}
\end{align}

In practice, the pipeline uploads the posterior samples to GraceDB, and gwcelery, the LVK's real-time alerts task system, performs and manages the actual EM-Bright calculation \cite{Chaudhary:2023vec, Magee:2021xdx}.

\subsection{Sky Localization}
GW sky localizations are represented as probability density distributions on the sky constructed in the HEALPix format \cite{Fernique}. Matched-filtering pipelines produce signal-to-noise ratio (SNR) time-series, which are processed by the BAYESTAR algorithm to produce fast sky localizations \cite{bayestar}. ML detection methods do not naturally produce SNR time-series, and so require an alternative. Two different estimators are used in real time to estimate sky localizations. First, an adaptive histogram estimator produces a rapid ($\lesssim 1$s) localization. Meanwhile, a slower ($\sim 20$s) but more robust kernel density estimator using the \texttt{ligo.skymap}\footnote{\url{https://git.ligo.org/lscsoft/ligo.skymap}} package is run in parallel. In Ref.~\cite{Marx:2025ioo}, we perform a detailed comparison of the two density estimation methods and find equal sky localization performance measured by searched area. However, the adaptive histogram estimator performs significantly worse as measured by searched volume (a three dimensional statistic) for low SNR events.  

\section{Software Design} \label{sec:software-arch}
The Aframe and AMPLFI deployment software is designed to minimize the latency to produce and upload each of the required data products described in Sec.~\ref{sec:data-products}. To enable this goal, software deployment consists of a main python process, and several python sub-processes that enable parallel execution of tasks. This allows tasks that are dependents of the same upstream data products to begin immediately. The main process is responsible for initial setup, data loading and processing, and model inference. After events are identified, a chain of sub-processes are triggered to produce derived data products. Processes are designed to begin work immediately once their data dependencies are available. In practice, this is implemented via a message passing system using python \texttt{multiprocessing.Queue} objects. Processes communicate data dependencies to others via a shared \texttt{multiprocessing.Queue}, and are then free to continue their own execution threads\footnote{We note the existence of several python based frameworks for distributed task orchestration, like Ray or Celery. We leave investigation of their utility to our problems to future work}. In Sec~\ref{sec:main-process} we outline the responsibilities of the main process, and in Sec~\ref{sec:sub-process} we describe each of the sub-processes. 

\subsection{Main Process} \label{sec:main-process}
When the pipeline is first launched, the main process will perform initial setup. This includes 

\begin{itemize}
    \item initializing Aframe and AMPLFI models from trained checkpoints and moving them to the GPU,
    \item initializing python sub-processes and \texttt{multiprocessing.Queue}'s for sharing data between processes,
    \item creating a GraceDB client instance for uploading data products to GraceDB,
    \item loading in a background model of noise events for estimating event significance,
    \item initializing buffers in GPU-memory for storing recent strain data and neural network outputs (see Sec.~\ref{sec:event-identification} and Sec.~\ref{sec:amplfi} for details)
\end{itemize}

The main process is the only process which communicates with GPU devices. 
After initial setup, the main process is responsible for the main strain data loading, processing, and validation loop. In addition, the main process performs inference with Aframe, and generates posterior samples with AMPLFI (see Sec.~\ref{sec:event-identification}). 

\subsection{Sub-processes} \label{sec:sub-process}
After event identification, python sub-processes are utilized to parallelize data product production. Python \texttt{multiprocessing.Queue} objects are used for message passing data dependencies between processes. Each sub-process operates by continually checking for available data in the queue, and immediately starts processing once available. A Python code snippet depicting this behavior is shown in Listing~\ref{lst:subprocess}. Below, we describe sub-process responsibilities, data products, and dependencies. Fig.~\ref{fig:subprocesses} provides an illustration of how sub-processes interact and depend upon each other.

\subsubsection{Event Creation}
After Aframe identifies an above threshold candidate, the main process will place the event's estimated coalescence time and FAR in the event creation sub-process queue. The event creation sub-process will then upload this information to GraceDB, creating a G-event. GraceDB returns to the client an identification number for the G-event. Uploading other data products to the G-event requires referencing this identification number. So, the event creation sub-process will place the identification number in the $p_{\mathrm{astro}}$ and AMPLFI sub-process queues.

\subsubsection{$p_{\mathrm{astro}}$}
Estimating $p_{\mathrm{astro}}$ only requires an events significance, so the $p_{\mathrm{astro}}$ sub-process is kicked off by the main process at the same time as event creation. However, uploading this information to GraceDB requires knowing the identification number of the Gevent. While $p_{\mathrm{astro}}$ estimation is ongoing, the event creation process is creating the Gevent, receiving its identification number from GraceDB, and placing it in the $p_{\mathrm{astro}}$ sub-processes queue. Once $p_{\mathrm{astro}}$ estimation is complete, the identification number is pulled from the queue $p_{\mathrm{astro}}$ is uploaded to the G-event. 

\subsubsection{Parameter Estimation}
The parameter estimation sub-process is responsible for calculating and uploading data products derived from posterior samples. After AMPLFI has finished sampling, the main process first populates a shared \texttt{multiprocessing.Array} object with the samples. Then, it adds additional event information like coalescence time, and available interferometers into the PE sub-process queue. 

Posterior samples produced by AMPLFI are standard scaled to be zero mean unit variance. So, the PE sub-process first descales the posterior samples to physically meaningful quantities using the mean and variance for each parameter estimated from AMPLFIs training data. Then, sky localization is estimated from the posterior samples using an adaptive histogram estimator. Meanwhile, the event creation process has shared the G-event ID with the PE sub-process. The sky map is uploaded to the G-event, along with the posterior samples, which are used by gwcelery to estimate EM-Bright quantities. 

\subsubsection{Authentication}
GraceDB utilizes the SciToken protocol for authentication \cite{DBLP:journals/corr/abs-1905-09816}. Successful communication with the GraceDB server requires keeping a fresh SciToken available. To do so, an additional sub-process continually refreshes the pipelines SciToken. The authentication sub-process does not directly communicate with the other processes.

\subsection{Deployment}
The pipeline is deployed and managed by a cronjob which launches the main process. The main process is executed inside an apptainer container environment. A shell-based monitoring script initializes the main process and restarts it upon exit for any reason. In some cases, sub-processes can fail  while the main process continues operating. When this occurs, the failed sub-process propagates its error to the main process via a python \texttt{\texttt{multiprocessing.Queue}} object. Upon receiving the error, the main process will  exit, killing each sub-process, and triggering a restart of the entire pipeline. Unfortunately, restarts cause incoming data to be skipped as the pipeline takes time to re-initialize and re-accumulate data buffers (see Sec.~\ref{sec:event-identification}). However, restarts are rare, typically occurring due to transient errors communicating with GraceDB, or issues with frame file delivery.

The computational resources used for deployment are an NVIDIA A30 GPU, and 8 CPU-cores. Both Aframe and AMPLFI models are able to fit in memory and perform inference on this GPU. The CPU cores are used primarily for data loading, post processing Aframe and AMPLFI outputs, and interacting with GraceDB.

\section{O4c Results} \label{sec:o4c}
Aframe and AMPLFI joined the O4c observing run on August 27th, 2025, operating until its completion on November 25th, 2025. During this period, 31 high significance public alerts were distributed via GCN. Of these, S251112cm was from a sub-solar mass (SSM) search, S251031dw was from a burst search, S250910b was from an early warning (EW) search, and S250908y, S250912f, S250917aq did not include either Hanford or Livingston as a trigger. This leaves 25 superevents to which Aframe and AMPLFI could have contributed. These events are listed in Table~\ref{table:o4c_events}. Of these 25 events, Aframe detected 23 of them. One non-detection, S250920bq, was detected only by SPIIR, with a FAR of $6.32 \times 10^{-8}$ Hz, just below the high significance threshold. The other, S250906ca, occurred during a power outage at the CIT cluster where Aframe is deployed, which caused issues with data delivery. 

Each Superevent is an aggregate of multiple G-events from potentially different pipelines which have estimated coalescence times that occur within some specified time interval. Each superevent has an associated preferred G-event, which is determined by the G-event which reports the largest SNR. Because Aframe does not estimate a SNR, G-events from other pipelines will be prioritized as the preferred event. However, if Aframe is the only G-event apart of a superevent, it will remain the preferred event until a G-event from another pipeline is submitted. When a G-event contains all of the data products necessary to distribute the preliminary alert, the preferred event is temporarily locked. The sky map corresponding to the preferred event is distributed with the alert. During O4c, there were 9 alerts where Aframe was the preferred at the time of the preliminary alert meaning the AMPLFI sky map was distributed. For all of these alerts, a matched filtering pipeline was made the preferred event after the preliminary alert was sent. Localization comparisons between AMPLFI and BAYESTAR for these 9 events where AMPLFI are shown in~\Cref{fig:S251103f,fig:S250830bp,fig:S251018bi,fig:S250830m,fig:S250901cb,fig:S250904br,fig:S251021u,fig:S251116en,fig:S251117dq}.

\section{Conclusions and Outlook}



Several avenues exist for improving the quality of this low latency deployment. Currently, if one sub-tasks fails, (e.g. due to an issue communicating with GraceDB), the whole system needs to restart, causing some data to not be analyzed. Integrating a distributed task orchestration framework like Ray or Celery will enable a scalable, fault-tolerant system. Failed sub-tasks can easily be retried without affecting other components of the system. Several contributors to latency can also be removed.
